# 1.2 用例文档

## 1. 参与者

| 参与者 | 描述 |
|--------|------|
| 游戏开发者 | 使用 OAsset 编辑器工具打包资源、配置系统 |
| 游戏客户端 | 运行时调用 OAsset API 加载和管理资源 |
| 资源服务器 | 托管 Bundle 文件和版本信息，提供热更新下载 |

## 2. 用例总览

```
游戏开发者                          游戏客户端
    |                                  |
    |-- UC-1 打包资源                    |-- UC-4 初始化资源系统
    |-- UC-2 强制重建资源                 |-- UC-5 加载资源
    |-- UC-3 查看 Bundle 信息            |-- UC-6 卸载 Bundle
    |                                  |-- UC-7 热更新资源
```

---

## 3. 用例详述

### UC-1 打包资源（默认策略）

- **参与者**：游戏开发者
- **前置条件**：Unity 编辑器已打开项目，待打包资源位于 `Assets/ToAB/` 目录
- **触发方式**：菜单 `OAsset > Build (Default Strategy)`

**主流程**：

1. 系统扫描 `Assets/ToAB/` 目录下所有支持的资源文件
2. 根据默认策略为每个资源生成 Bundle 名称并设置到 AssetImporter
3. 删除并重建输出目录 `Assets/StreamingAssets/AB/`
4. 调用 Unity BuildPipeline 构建 AssetBundle（LZ4 压缩）
5. 生成 `version.txt`（格式：`yyyy.M.d.HHmmss`）
6. 生成 `<版本号>.manifest.json`，包含所有资源和 Bundle 的元信息
7. 清理 Unity 自动生成的 `.manifest` 文件和主 manifest bundle
8. 刷新 AssetDatabase

**后置条件**：`Assets/StreamingAssets/AB/` 目录包含 version.txt、manifest.json 和所有 Bundle 文件

---

### UC-2 强制重建资源

- **参与者**：游戏开发者
- **前置条件**：同 UC-1
- **触发方式**：菜单 `OAsset > Build (Force Rebuild)`

**主流程**：与 UC-1 相同，但构建时附加 `ForceRebuildAssetBundle` 标志，忽略增量缓存，重新构建所有 Bundle。

**适用场景**：资源出现异常、需要完全清理重建时使用。

---

### UC-3 查看 Bundle 信息

- **参与者**：游戏开发者
- **触发方式**：菜单 `OAsset > Show Bundle Info`

**主流程**：

1. 获取所有已设置的 Bundle 名称
2. 对每个 Bundle，获取其包含的资源列表
3. 在 Console 中输出 Bundle 名称及其包含的资源路径

---

### UC-4 初始化资源系统

- **参与者**：游戏客户端
- **前置条件**：已创建 `OAssetConfig` 配置资产
- **API**：`OAsset.Instance.InitializeAsync(config)`

**主流程**：

1. 检查是否已初始化，避免重复初始化
2. 检查缓存目录是否存在

**分支A - 首次启动**（缓存目录不存在）：

3a. 创建缓存目录结构（含 `BundleFiles/`）
4a. 从 StreamingAssets 读取 `version.txt` 和 manifest.json
5a. 遍历 manifest 中的每个 Bundle，从 StreamingAssets 读取，校验文件大小和 CRC32 后写入缓存目录
6a. 将 version.txt 和 manifest.json 复制到缓存根目录

**分支B - 非首次启动**（缓存目录已存在）：

3b. 若配置了 `serverUrl`，下载远程 `version.txt`
4b. 比较远程版本与本地版本
5b. 版本不一致时，下载新的 manifest.json 并更新本地文件，同时删除旧的 manifest 文件

**汇合**：

7. 读取本地 version.txt 和 manifest.json
8. 反序列化 manifest，构建资源路径查询字典
9. 标记初始化完成

**后置条件**：`OAsset.Instance.IsInitialized == true`，可以开始加载资源

**异常流程**：
- 网络不可用时，使用本地已缓存的清单继续初始化
- 清单解析失败时，记录日志并使用本地缓存

---

### UC-5 加载资源

- **参与者**：游戏客户端
- **前置条件**：已完成初始化（UC-4）
- **API**：`OAsset.Instance.LoadAssetAsync<T>(assetPath)`

**主流程**：

1. （编辑器模式）尝试通过 `AssetDatabase.LoadAssetAtPath<T>()` 直接加载，成功则返回
2. 在 manifest 中查找 assetPath 对应的 AssetInfo
3. 通过 AssetInfo.BundleID 获取所属 BundleInfo，从 BundleInfo.DependBundleIDs 获取依赖列表
4. 对每个依赖 Bundle 执行加载（递归）
5. 加载主 Bundle

**Bundle 加载子流程**（`EnsureBundleLoaded`）：

5a. 检查 `_loadedBundles` 缓存，已加载则跳过
5b. 检查 `_loadingBundles`，有正在加载的任务则 await 该任务
5c. 使用 `UniTask.Lazy` 创建加载任务并存入 `_loadingBundles`，消除竞态窗口
5d. 检查本地磁盘缓存是否存在（`IsBundleCached`）
5e. 不存在则从服务器下载（含大小 + CRC32 校验）
5f. 通过 `AssetBundle.LoadFromFileAsync` 从缓存的 `__data` 文件加载
5g. 加载成功，存入 `_loadedBundles` 缓存

6. 从已加载的 Bundle 中加载目标资源 `bundle.LoadAssetAsync<T>(assetPath)`
7. 返回资源对象

**异常流程**：
- 资源路径未在 manifest 中找到：记录错误日志，返回 null
- Bundle 加载失败：删除损坏的缓存目录，记录错误日志，返回 null

---

### UC-6 卸载 Bundle

- **参与者**：游戏客户端

**UC-6a 卸载单个 Bundle**：
- **API**：`OAsset.Instance.UnloadBundle(fileHash, unloadAllObjects)`
- 从 `_loadedBundles` 中移除并调用 `AssetBundle.Unload()`

**UC-6b 卸载所有 Bundle**：
- **API**：`OAsset.Instance.UnloadAllBundles(unloadAllObjects)`
- 遍历 `_loadedBundles` 全部卸载，清空 `_loadedBundles` 和 `_loadingBundles`

---

### UC-7 热更新资源

- **参与者**：游戏客户端、资源服务器
- **前置条件**：配置了 `serverUrl`，服务器已部署新版本资源

**主流程**：

1. 初始化时（UC-4 分支B），客户端下载服务器 `version.txt`
2. 发现版本号不一致，下载新的 `<版本号>.manifest.json`
3. 更新本地 version.txt 和 manifest.json，删除旧的 manifest 文件
4. 后续加载资源时（UC-5），若 Bundle 的 MD5 对应的缓存不存在，自动从服务器下载

**说明**：热更新采用懒加载策略——初始化时只更新清单，Bundle 文件在实际需要时才下载。

---

## 4. 用例关系

```
UC-4 初始化资源系统
    ├── <<include>> 首次启动初始化（从 StreamingAssets 复制，含 CRC 校验）
    └── <<include>> 版本检查与清单更新（含旧清单清理）

UC-5 加载资源
    ├── <<include>> Bundle 依赖加载（从 BundleInfo 获取依赖）
    ├── <<include>> Bundle 缓存查询
    └── <<extend>>  Bundle 下载（缓存未命中时）

UC-7 热更新资源
    ├── <<include>> UC-4 初始化（版本比对触发）
    └── <<include>> UC-5 加载（按需下载新 Bundle）
```
