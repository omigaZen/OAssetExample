# OAsset 待修复问题与建议

以下问题来自代码审查，按优先级排列，可在后续版本中逐步改进。

---

## 严重

### 1. 单例不支持重置，热重载/测试困难

**文件**: `Runtime/OAsset.cs:12-13`

`_instance` 一旦创建无法销毁。如果需要切换配置（如切换服务器环境）或在单元测试中重置状态，目前无法做到。

**建议**: 提供 `Destroy()` 方法，将 `_instance` 置空并清理所有已加载的 Bundle。

### 2. 首次安装全量复制 Bundle 会长时间阻塞

**文件**: `Runtime/OAsset.cs:253-271`（`InitFromStreamingAssets`）

所有 Bundle 在初始化时全量复制并校验 CRC32，大项目（几百个 Bundle）会造成首次启动长时间卡顿。

**建议**: 改为延迟复制（按需从 StreamingAssets 加载），或增加进度回调让 UI 层展示加载进度。

---

## 中等

### 3. EnsureBundleLoaded 并发安全前提应明确

**文件**: `Runtime/OAsset.cs:167-175`

`ContainsKey` 和 `TryGetValue` 两次检查之间没有锁保护。当前在 Unity 主线程模型下是安全的（UniTask 默认在主线程调度），但如果未来引入多线程调度则可能出现重复加载。

**建议**: 在方法注释中明确标注「此方法假设在 Unity 主线程上调用」，或加入简单的同步保护。

### 4. 同步文件 I/O 在主线程执行

**文件**: `Runtime/OAsset.cs:57-58, 71-72, 98`

`File.ReadAllText`、`File.WriteAllText`、`File.WriteAllBytes` 在主线程上同步执行。小文件影响不大，但 manifest JSON 可能较大。

**建议**: 使用 `UniTask.Run` 在后台线程执行文件 I/O，或使用异步文件 API。

### 5. Editor 模式无法测试真实 Bundle 加载流程

**文件**: `Runtime/OAsset.cs:114-120`

Editor 模式下 `AssetDatabase.LoadAssetAtPath` 成功就直接返回，完全跳过了 manifest 和 Bundle 逻辑，导致 Editor 下永远无法测试热更新流程。

**建议**: 在 `OAssetConfig` 中增加 `forceUseBundles` 开关，Editor 下也可走 Bundle 路径以便调试。

---

## 轻微

### 6. UnloadBundle API 暴露 fileHash 参数，对调用者不友好

**文件**: `Runtime/OAsset.cs:145`

外部调用者通常只知道资源路径或 Bundle 名，不知道内部 MD5 哈希。

**建议**: 提供基于 `bundleName` 或 `assetPath` 的卸载接口，内部查找对应的 `fileHash`。

### 7. 缺少 CancellationToken 支持

所有异步方法（`InitializeAsync`、`LoadAssetAsync`、`DownloadFile` 等）都不接受 `CancellationToken`，无法在场景切换或应用退出时取消正在进行的操作。

**建议**: 为所有公开的异步方法增加可选的 `CancellationToken` 参数，并在内部传递给 UniTask 和 UnityWebRequest。

### 8. UnloadAllBundles 与正在进行的加载任务状态不一致

**文件**: `Runtime/OAsset.cs:159`

如果有 Bundle 正在下载/加载中，清空 `_loadingBundles` 字典后，正在执行的加载任务会继续运行并将结果存入已被清空的 `_loadedBundles`，产生状态不一致。

**建议**: 配合 `CancellationToken` 取消所有进行中的加载任务，或在加载完成时检查是否已被卸载。
