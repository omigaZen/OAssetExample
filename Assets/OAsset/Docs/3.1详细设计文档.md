# 3.1 详细设计文档

## 1. 编辑器模块

### 1.1 OAssetBuilder（打包工具）

**职责**：提供 AssetBundle 构建流水线，生成 Bundle 文件和资源清单。

**类概览**：

```csharp
namespace OAsset.Editor
{
    public static class OAssetBuilder
    {
        private const string OutputDir = "Assets/StreamingAssets/AB";
        private const string VersionFileName = "version.txt";
    }
}
```

#### 1.1.1 构建流程 `BuildInternal(bool forceRebuild)`

```
BuildInternal(forceRebuild)
│
├─ 1. ScanAssets(strategy)
│     遍历 strategy.GetScanDirectories() 下所有文件
│     按 strategy.GetAssetFilters() 过滤扩展名
│     返回去重后的资源路径列表
│
├─ 2. 设置 Bundle 名称
│     对每个资源路径调用 strategy.GetBundleName()
│     通过 AssetImporter.SetAssetBundleNameAndVariant() 设置
│
├─ 3. AssetDatabase.RemoveUnusedAssetBundleNames()
│     清理不再使用的 Bundle 名称
│
├─ 4. 清理并重建输出目录
│     Directory.Delete(OutputDir, true)
│     Directory.CreateDirectory(OutputDir)
│
├─ 5. BuildPipeline.BuildAssetBundles()
│     选项: ChunkBasedCompression（LZ4）
│     可选: ForceRebuildAssetBundle
│     目标平台: EditorUserBuildSettings.activeBuildTarget
│
├─ 6. 生成版本号
│     格式: DateTime.Now.ToString("yyyy.M.d.HHmmss")
│     写入: OutputDir/version.txt
│
├─ 7. GenerateManifest(strategy, assetPaths, buildManifest)
│     生成: OutputDir/<版本号>.manifest.json
│
├─ 8. CleanupUnityManifests()
│     删除所有 .manifest 文件
│     删除与输出目录同名的主 manifest bundle
│
└─ 9. AssetDatabase.Refresh()
```

#### 1.1.2 清单生成 `GenerateManifest()`

```
GenerateManifest(strategy, assetPaths, buildManifest)
│
├─ 1. 构建 bundleName → index 映射
│     从 buildManifest.GetAllAssetBundles() 获取所有 Bundle 名
│     为每个 Bundle 分配连续整数索引
│
├─ 2. 填充 BundleList
│     对每个 Bundle:
│     ├─ BundleName: Bundle 文件名
│     ├─ FileSize:   new FileInfo(path).Length
│     ├─ FileCRC:    FileDownloader.ComputeFileCRC32(path)
│     ├─ FileHash:   ComputeFileMD5(path)  // MD5 哈希
│     ├─ UnityCRC:   BuildPipeline.GetCRCForAssetBundle(path)
│     └─ DependBundleIDs: 将依赖 Bundle 名转换为索引数组
│
├─ 3. 填充 AssetList
│     对每个资源路径:
│     ├─ AssetPath: 原始路径
│     └─ BundleID:  所属 Bundle 的索引
│
└─ 4. 序列化为 JSON 并写入文件
```

#### 1.1.3 Unity 产物清理 `CleanupUnityManifests()`

Unity 的 `BuildPipeline.BuildAssetBundles()` 会自动生成以下额外文件，OAsset 不需要它们：

- `*.manifest` 文件（每个 Bundle 一个文本格式的 manifest）
- 与输出目录同名的主 manifest bundle（如 `AB/AB`）

此方法在构建后删除这些文件，只保留 Bundle 文件本身和 OAsset 自己生成的清单。

### 1.2 IAssetBundleStrategy（策略接口）

```csharp
public interface IAssetBundleStrategy
{
    string GetBundleName(string assetPath);
    string[] GetScanDirectories();
    string[] GetAssetFilters();
}
```

策略接口将「哪些资源需要打包」和「如何命名 Bundle」的决策从构建流程中解耦，允许替换不同的分包策略。

### 1.3 DefaultStrategy（默认策略）

| 配置项 | 值 |
|--------|-----|
| 扫描目录 | `Assets/ToAb` |
| 支持的扩展名 | `.prefab`, `.png`, `.jpg`, `.jpeg`, `.tga`, `.bmp`, `.mat`, `.anim`, `.controller`, `.asset`, `.unity`, `.bytes`, `.txt`, `.json`, `.wav`, `.mp3`, `.ogg`, `.shader`, `.compute`, `.ttf`, `.otf`, `.fontsettings`, `.mesh`, `.fbx`, `.obj`（共 27 种） |

## 2. 运行时模块

### 2.1 OAsset（资源管理器单例）

**职责**：资源系统的入口，管理初始化、加载、缓存、卸载全生命周期。

**单例模式**：

```csharp
private static OAsset _instance;
public static OAsset Instance => _instance ??= new OAsset();
```

懒加载单例，首次访问时创建。非 MonoBehaviour，纯 C# 对象。

#### 2.1.1 初始化流程 `InitializeAsync(OAssetConfig config)`

```
InitializeAsync(config)
│
├─ Guard: 已初始化则直接返回
│
├─ 设置 _config, _cacheRoot
│
├─ 判断: Directory.Exists(_cacheRoot)?
│  │
│  ├─ 否（首次启动）→ InitFromStreamingAssets()
│  │   ├─ 创建 BundleFiles/ 目录
│  │   ├─ 读取 StreamingAssets 中的 version.txt
│  │   ├─ 读取 StreamingAssets 中的 <ver>.manifest.json
│  │   ├─ 遍历 BundleList:
│  │   │   ├─ 读取 StreamingAssets 中的 Bundle 文件（字节数组）
│  │   │   ├─ 校验文件大小和 CRC32
│  │   │   └─ FileDownloader.WriteBundleToCache(bytes, bundle, cacheRoot)
│  │   └─ 写入 version.txt 和 manifest.json 到缓存根目录
│  │
│  └─ 是 + serverUrl 非空 → 版本检查
│      ├─ FileDownloader.DownloadText(serverUrl + "version.txt")
│      ├─ 比较远程版本与本地版本
│      └─ 不同 → 下载新 manifest，更新本地 version.txt，删除旧 manifest
│
├─ 读取本地 version.txt 和 manifest.json
├─ _manifest = Manifest.FromJson(json)  // 内部自动 BuildLookup()
└─ _initialized = true
```

**平台差异处理**：

| 操作 | Android | 其他平台 |
|------|---------|----------|
| StreamingAssets URL | `Application.streamingAssetsPath + "/" + subPath`（jar:file://协议） | `Path.Combine()` |
| 读取文本 | `UnityWebRequest.Get()` + `downloadHandler.text`（独立 async 方法） | `File.ReadAllText()`（返回 `UniTask.FromResult`，无 async 警告） |
| 读取字节 | `UnityWebRequest.Get()` + `downloadHandler.data`（独立 async 方法） | `File.ReadAllBytes()`（返回 `UniTask.FromResult`，无 async 警告） |

#### 2.1.2 资源加载 `LoadAssetAsync<T>(string assetPath)`

```
LoadAssetAsync<T>(assetPath)
│
├─ [UNITY_EDITOR] AssetDatabase.LoadAssetAtPath<T>()
│   成功则直接返回
│
├─ _manifest.TryGetAsset(assetPath, out assetInfo)
│   失败 → 抛出 FileNotFoundException
│
├─ 获取 mainBundle = _manifest.BundleList[assetInfo.BundleID]
│
├─ 加载依赖 Bundle（从 BundleInfo 获取依赖）:
│   foreach (depID in mainBundle.DependBundleIDs)
│       await EnsureBundleLoaded(_manifest.BundleList[depID])
│
├─ await EnsureBundleLoaded(mainBundle)
│
├─ bundle = _loadedBundles[mainBundle.FileHash]
│
└─ return bundle.LoadAssetAsync<T>(assetPath)
```

#### 2.1.3 Bundle 加载 `EnsureBundleLoaded(BundleInfo bundle)`

这是防重复加载的核心方法，使用 `UniTask.Lazy` 消除竞态窗口：

```
EnsureBundleLoaded(bundle)
│
├─ _loadedBundles 中已存在? → return（已加载）
│
├─ _loadingBundles 中已存在? → await 已有 Task → return（防并发重复）
│
├─ 使用 UniTask.Lazy 创建延迟 Task
├─ 存入 _loadingBundles[bundle.FileHash]（Task 此时尚未开始执行）
├─ await lazy.Task（开始执行并等待完成）
└─ finally: 从 _loadingBundles 中移除
```

**竞态修复说明**：

旧版本中，`LoadBundleInternal` 在调用时立即开始执行，存入 `_loadingBundles` 之前可能已经 yield。
使用 `UniTask.Lazy` 后，Task 在首次 await 时才开始执行，确保存入字典后才运行，消除了检查与插入之间的竞态窗口。

**并发去重机制**：

```
时序示例:
  T1: LoadAsset("A.prefab") → EnsureBundleLoaded(bundleX)
      → _loadingBundles 无 → 创建 Lazy Task, 存入 _loadingBundles
  T2: LoadAsset("B.prefab") → EnsureBundleLoaded(bundleX)  [依赖同一 Bundle]
      → _loadingBundles 有 → await 同一 Task
  T3: bundleX 加载完成 → T1 和 T2 同时继续
```

`Task`/`UniTask` 可以被多次 await，已完成的 Task 再次 await 时立即返回结果。

#### 2.1.4 Bundle 内部加载 `LoadBundleInternal(BundleInfo bundle)`

```
LoadBundleInternal(bundle)
│
├─ FileDownloader.IsBundleCached(cacheRoot, bundle.FileHash)?
│   └─ 否 → await FileDownloader.DownloadBundle(bundle, cacheRoot, serverUrl, timeout)
│
├─ cachePath = FileDownloader.GetBundleCachePath(cacheRoot, bundle.FileHash)
├─ dataPath = Path.Combine(cachePath, "__data")
│
├─ ab = await AssetBundle.LoadFromFileAsync(dataPath)
│
├─ ab == null?
│   ├─ 是 → Directory.Delete(cachePath, true)  // 删除损坏缓存
│   │       throw Exception
│   └─ 否 → _loadedBundles[bundle.FileHash] = ab
│
└─ return ab
```

#### 2.1.5 卸载

**UnloadBundle(fileHash, unloadAllObjects)**：
```csharp
if (_loadedBundles.TryGetValue(fileHash, out var bundle))
{
    bundle.Unload(unloadAllObjects);
    _loadedBundles.Remove(fileHash);

```

**UnloadAllBundles(unloadAllObjects)**：
```csharp

    bundle.Unload(unloadAllObjects);

```

### 2.2 Manifest（清单数据模型）

**查询机制**：

```
TryGetAsset(assetPath)
│
├─ _assetLookup 为空? → BuildLookup()
│   遍历 AssetList，构建 AssetPath → AssetInfo 字典
│
└─ return _assetLookup.TryGetValue(assetPath, out info)
```

`BuildLookup()` 在 `_assetLookup` 为 null 时延迟构建字典，避免反序列化后未使用时的不必要开销。`FromJson()` 内部自动调用 `BuildLookup()`。

**序列化**：使用 Unity 的 `JsonUtility`，字段直接映射 JSON 键名。

### 2.3 FileDownloader（文件下载与缓存 IO）

**职责**：处理所有网络下载、磁盘缓存读写和数据校验。

纯静态类，无状态。

#### 2.3.1 通用重试机制

所有下载方法通过通用的 `RetryAsync<T>` 和 `RetryAsync` 方法实现重试逻辑，消除重复代码：

```csharp
private static async UniTask<T> RetryAsync<T>(Func<UniTask<T>> action, string description)
{
    Exception lastError = null;
    for (int attempt = 0; attempt < MaxRetries; attempt++)
    {
        if (attempt > 0)
            await UniTask.Delay(TimeSpan.FromSeconds(Math.Pow(2, attempt)));
        try { return await action(); }
        catch (Exception e) { lastError = e; Debug.LogWarning(...); }
    }
    throw new Exception($"[OAsset] {description} failed after {MaxRetries} attempts", lastError);
}
```

同时提供无返回值的 `RetryAsync(Func<UniTask>, string)` 重载，供 `DownloadFile` 等不返回值的方法使用。

```
最大重试次数: 3
退避策略: 指数退避，等待 2^attempt 秒
  第1次失败 → 等待 2 秒
  第2次失败 → 等待 4 秒
  第3次失败 → 抛出异常
```

#### 2.3.2 Bundle 下载流程 `DownloadBundle()`

```
DownloadBundle(bundle, cacheRoot, serverUrl, timeout, onProgress)
│
├─ url = serverUrl + bundle.BundleName
├─ tmpPath = Path.Combine(cacheRoot, bundle.BundleName + ".tmp")
│
├─ await DownloadFile(url, tmpPath, timeout, onProgress)
│
├─ 校验文件大小:
│   new FileInfo(tmpPath).Length == bundle.FileSize?
│   └─ 否 → 删除 tmp, 抛出异常
│
├─ 校验 CRC32:
│   FileDownloader.ComputeFileCRC32(tmpPath) == bundle.FileCRC?
│   └─ 否 → 删除 tmp, 抛出异常
│
└─ WriteBundleToCache(tmpPath, bundle, cacheRoot)
    ├─ 目标目录: BundleFiles/{MD5[0:2]}/{MD5}/
    ├─ File.Move(tmpPath → __data)
    └─ File.WriteAllText(__info, "{CRC32},{FileSize}")
```

#### 2.3.3 CRC32 实现

使用预计算的 256 项查找表，在静态构造函数中初始化，标准多项式 `0xEDB88320`（反射多项式）：

```csharp
private static readonly uint[] _crc32Table;  // 256 项查找表，静态构造函数中初始化

static FileDownloader()
{
    _crc32Table = new uint[256];
    for (uint i = 0; i < 256; i++)
    {
        uint crc = i;
        for (int j = 0; j < 8; j++)
            crc = (crc >> 1) ^ (0xEDB88320 & ~((crc & 1) - 1));
        _crc32Table[i] = crc;
    }
}

public static uint ComputeCRC32(byte[] data)
{
    uint crc = 0xFFFFFFFF;
    foreach (byte b in data)
        crc = (crc >> 8) ^ _crc32Table[(crc ^ b) & 0xFF];
    return ~crc;
}
```

提供三个版本：
- `ComputeCRC32(byte[])` — 用于内存中的数据（如 StreamingAssets 复制时的校验）
- `ComputeCRC32(Stream)` — 使用 8KB 缓冲区流式读取，避免大文件一次性加载到内存
- `ComputeFileCRC32(string)` — 打开文件流后调用流式版本

#### 2.3.4 缓存路径计算

```csharp
public static string GetBundleCachePath(string cacheRoot, string fileHash)
{
    return Path.Combine(cacheRoot, "BundleFiles", fileHash.Substring(0, 2), fileHash);
}
// 示例: GetBundleCachePath("/cache", "e99a18c428cb38d5f260853678922e03")
//    → "/cache/BundleFiles/e9/e99a18c428cb38d5f260853678922e03"
```

#### 2.3.5 缓存检测

```csharp
public static bool IsBundleCached(string cacheRoot, string fileHash)
{
    string path = GetBundleCachePath(cacheRoot, fileHash);
    return File.Exists(Path.Combine(path, "__data"));
}
```

仅检查 `__data` 文件是否存在，不做内容校验（性能考量）。

## 3. 关键设计决策

### 3.1 MD5 作为缓存键

使用文件内容的 MD5 哈希（而非文件名）作为缓存目录名：
- **优势**：内容变化时 MD5 自然变化，旧缓存自动失效，无需显式清理
- **优势**：相同内容的 Bundle 即使文件名不同也只缓存一份
- **代价**：构建时需计算 MD5，运行时无法从文件名直接定位缓存

### 3.2 双重校验（CRC32 + FileSize）

下载完成后和首次从 StreamingAssets 复制时同时校验文件大小和 CRC32：
- FileSize 校验快速排除明显的传输错误（截断、重复写入）
- CRC32 校验检测内容级别的数据损坏

### 3.3 懒加载 Bundle

热更新时只下载清单，不预下载 Bundle 文件。Bundle 在实际需要加载资源时才下载：
- **优势**：减少初始化时间和流量消耗
- **代价**：首次加载某资源时可能需要等待下载

### 3.4 编辑器模式直通

`LoadAssetAsync<T>` 在 `UNITY_EDITOR` 宏下优先使用 `AssetDatabase.LoadAssetAtPath<T>()`，开发时无需打包即可运行。

### 3.5 并发去重

通过 `_loadingBundles` 字典存储进行中的加载 Task，多个请求加载同一 Bundle 时共享同一个 Task，避免重复加载和 Unity 报错（同一 Bundle 不能同时加载两次）。使用 `UniTask.Lazy` 确保 Task 在存入字典后才开始执行，消除竞态窗口。
